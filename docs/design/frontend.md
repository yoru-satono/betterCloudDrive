# Web 前端概要设计

## 职责

Web 前端是浏览器用户的主入口，负责认证、文件浏览、上传下载、分享、回收站、收藏、标签、个人资料、WebDAV 设置、管理后台和日志中心。它也承担 Tauri 桌面端的 UI 基座，但不能依赖桌面原生能力才能在浏览器中运行。

## 不负责的内容

- 不做最终权限判断。
- 不直接访问数据库、Redis 或对象存储。
- 不保存必须跨设备一致的业务事实。
- 不在浏览器端执行文件夹直下逻辑，文件夹直下属于 Tauri 能力。

## 路由与权限

路由分三组：

- 公开路由：登录、注册、忘记密码、公开分享。
- 普通登录路由：文件、回收站、分享、收藏、标签、资料、桌面设置。
- 管理员路由：管理面板、用户管理、用户文件、日志中心。

权限处理：

1. 路由守卫检查本地 access token。
2. 需要管理员时调用或复用当前用户信息。
3. 未登录用户跳转登录页。
4. 已登录用户访问 guest 页面时跳转文件页。
5. 桌面专属设置页在浏览器中不可访问。

注意：前端守卫只提升体验，后端仍必须做完整权限校验。

## API 层设计

`src/api/client.ts` 负责基础 Axios 实例、token 注入、错误处理和刷新策略。功能 API 按领域拆分：

- `auth.ts`：认证和用户资料。
- `files.ts`：文件操作和搜索。
- `upload.ts` / `resumableUpload.ts`：网页端上传。
- `download.ts`：网页端 ticket 下载。
- `shares.ts`：分享管理和公开访问。
- `admin.ts`：管理后台。
- `desktop*.ts`：Tauri command 封装。

地址解析由 `src/config/runtime.ts` 处理：

- Web 默认使用同域 `/api/v1`。
- Tauri 可使用桌面构建参数或用户配置。
- 本地开发可通过 Vite proxy 指向后端。

## 状态管理

| Store | 职责 | 持久化 |
|------|------|------|
| `auth` | token、当前用户、管理员状态 | localStorage token |
| `files` | 当前目录、面包屑、文件列表、搜索结果 | 页面状态 |
| `upload` | 网页端上传任务、分片进度、暂停恢复 | localStorage 续传记录 |
| `transfer` | 桌面上传/下载统一队列 | Tauri `transfers.json` |
| `preview` | 当前预览文件和弹窗状态 | 页面状态 |
| `ui` | 全局 UI 状态 | 页面状态 |

## 文件浏览设计

文件页围绕“当前目录”组织：

1. 根据路由或状态确定 `parentId`。
2. 请求后端分页文件列表。
3. 更新面包屑和当前目录。
4. 用户执行新建、重命名、移动、复制、删除、收藏、标签等动作。
5. 动作成功后刷新当前目录或局部更新。

搜索不应污染“全部文件”的当前目录列表。搜索结果应作为独立候选层展示；点击文件打开预览，点击文件夹进入对应目录并同步当前目录。

## 上传设计

网页端上传基于后端分片 API：

1. 用户选择一个或多个文件。
2. 前端计算文件大小、名称和可选 MD5。
3. 尝试秒传。
4. 秒传失败则初始化上传会话。
5. 按并发策略上传分片。
6. 暂停时停止调度新分片并保留本地会话记录。
7. 恢复时调用 status，只补传缺失分片。
8. complete 后刷新文件列表。

多个文件上传必须经过队列调度，不能让每个文件无限制并发。UI 需要展示每个文件的状态、进度、失败原因和可恢复动作。

## 下载设计

网页端下载有浏览器限制，因此分两类：

- 单文件：创建下载 ticket，使用浏览器下载能力，后端 Range 支持暂停继续。
- 文件夹：创建 ZIP ticket，服务端生成可复用 ZIP 缓存，缓存按最后下载时间保留。

浏览器通常不能可靠控制用户最终保存位置，也不能稳定写入目录结构，因此“直接下载文件夹”只在 Tauri 客户端中实现。

## 管理后台设计

管理后台功能：

- 系统统计。
- 用户列表、状态修改、配额调整。
- 用户文件浏览。
- 审计日志查询。
- 系统日志摘要查询。
- Grafana 跳转。

日志中心设计：

- 审计日志表格从数据库审计记录读取，可展开 JSON 详情。
- 系统日志只显示摘要，不在前端承载完整日志平台能力。
- Grafana 链接需要先创建后端短期会话。

## 桌面端复用策略

前端在运行时判断是否处于 Tauri：

- 浏览器：使用 Web API 和网页端上传下载。
- Tauri：调用 `desktopUpload`、`desktopDownload`、`desktopSettings`、`desktopTransfer`。

桌面增强能力不能破坏浏览器路径。新功能如果需要本地文件系统，应提供浏览器降级或隐藏入口。

## UI 组件约定

- 文件列表和文件网格复用同一套文件操作。
- 右键菜单应与普通文件列表行为一致。
- 传输队列应显示上传和下载两个标签页。
- 数字显示使用等高数字样式，避免进度和大小跳动。
- 错误提示需要可操作，至少说明失败动作和下一步。

## 失败处理

| 场景 | 前端处理 |
|------|------|
| access token 过期 | 尝试 refresh，失败后回登录页 |
| 上传分片失败 | 标记任务错误，可恢复时保留会话 |
| 下载 ticket 失效 | 重新创建 ticket |
| 分享密码错误 | 保持页面并提示重新输入 |
| 管理员权限失效 | 跳转普通文件页或显示无权限 |
| 后端返回 requestId | 错误弹窗可展示或复制 requestId |

## 测试关注点

- 登录注册和 token 刷新。
- 文件列表、搜索和右键菜单。
- 多文件上传队列、暂停、恢复、失败重试。
- 单文件和文件夹下载 ticket。
- 分享创建、密码读取、公开访问。
- 管理员用户、配额、日志中心。
- Tauri 运行时下桌面能力入口。

## 扩展建议

- 新业务先补 API 封装和类型，再补 store 和页面。
- 公共 UI 行为放入 composable，避免页面复制。
- 桌面专属能力隔离在 `desktop*` API 中。
- 路由新增权限时同步补 E2E。
